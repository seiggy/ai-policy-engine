## Work Summary

Created two DLP-enabled APIM policy variants to support Purview DLP content-check enforcement alongside existing precheck-only base policies. Offers customers choice between baseline (auth/quota only) and DLP-enabled (auth/quota + content blocking) policies.

## Decisions Made

- **Policy Variants:** Created 2 new DLP policy files (subscription-key + entra-jwt variants)
- **Content-Check Placement:** After precheck succeeds, after routing logic, before backend forwarding
- **Fail-Open Strategy:** Uses `ignore-error="true"` on send-request; transient failures don't block requests
- **HTTP 451 Response:** Explicit block response only; all other statuses/failures proceed
- **Error Format:** Mimics Azure OpenAI content_filter response for client compatibility
- **Timeout:** 10 seconds (matches precheck timeout)
- **Authentication:** APIM managed identity token (consistent with precheck)
- **Base Policies Unchanged:** No modifications to existing subscription-key-policy.xml or entra-jwt-policy.xml

## Files Created

1. `policies/subscription-key-policy-dlp.xml` — Subscription key auth + DLP content-check
2. `policies/entra-jwt-policy-dlp.xml` — Entra JWT auth + DLP content-check

## Files Unchanged

1. `policies/subscription-key-policy.xml` — Base policy remains identical
2. `policies/entra-jwt-policy.xml` — Base policy remains identical
