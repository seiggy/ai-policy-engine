## Sydnor: APIM DLP Policy Variants

### Work Completed
- Created `policies/subscription-key-policy-dlp.xml` (subscription key auth + DLP)
- Created `policies/entra-jwt-policy-dlp.xml` (Entra JWT auth + DLP)
- Maintained base policies unchanged for non-DLP customers
- Implemented fail-open content-check with HTTP 451 blocking

### Key Design Decisions
- **Content-Check Placement:** After precheck + routing, before backend forwarding
- **Fail-Open Strategy:** Transient failures don't block requests; only HTTP 451 blocks
- **Backward Compatibility:** Existing deployments continue using base policies unchanged

### Key Files Modified/Created
- Created: `policies/subscription-key-policy-dlp.xml`
- Created: `policies/entra-jwt-policy-dlp.xml`
- Unchanged: `policies/subscription-key-policy.xml`
- Unchanged: `policies/entra-jwt-policy.xml`

### Learnings
- Four policy variants to maintain (base + DLP per auth type) creates future maintenance burden
- Fail-open strategy prioritizes availability over strict DLP enforcement for transient failures
- Policy selection guide simplifies customer decision (use base if no DLP, use -dlp suffix if DLP required)
